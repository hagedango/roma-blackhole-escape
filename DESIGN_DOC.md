## 0. プロジェクト概要

| 項目 | 内容 |
|---|---|
| タイトル（仮） | ロマ子様のブラックホール・エスケープ |
| ジャンル | 固定画面・引力回避アクションゲーム |
| プラットフォーム | ブラウザ（PC / スマホ両対応） |
| 技術構成 | HTML + CSS + JavaScript（Vanilla JS、外部ライブラリ・ビルドツール不要） |
| 配布形態 | フォルダをダブルクリックして `index.html` を開けばそのまま遊べること（サーバー不要、npm install 不要） |
| 想定プレイ時間 | 1プレイ 30秒〜2分程度のショートゲーム |
| ターゲットユーザー | ロマ子様コミュニティのメンバー（非エンジニアも含む一般ユーザー） |

### ゲームコンセプト（一言で）
画面中央付近に出現したブラックホールが、時間経過とともに**プレイヤーを引き寄せながら拡大**していく。プレイヤー（ロマ子様）は飛んでくる隕石（障害物）を避けつつ、ブラックホールの引力に逆らって**できるだけ長く生き延びる**。スコアは「生存時間＋スターのかけら収集ボーナス」の合計。

### ゲームデザインの狙い（実装者向けメモ）
- ただ避けるだけの受け身ゲームにしないため、**スターのかけら（収集アイテム）をブラックホール付近に出やすく**配置し、「安全に逃げるか、危険を冒して稼ぐか」のリスク＆リターンを作る。
- **ダッシュ（緊急回避）**により「吸い込まれそう→脱出！」の爽快感を作る。
- **ロマ子様のセリフ演出**でキャラクターゲームとしての愛嬌を出す。
- **吸い込みモード（引力サージ）**を周期的に発生させ、ゲーム後半が単調にならないよう緊張と緩和のリズムを作る。

## 1. 画面フローと遷移

3つの画面を持ち、すべて1つの `index.html` 内で `display: none/block` を切り替える「シングルページ・ステートマシン」方式で実装する（画面遷移にページ遷移やルーティングライブラリは使わない）。

```
[タイトル画面]
   │ 難易度を選択（EASY / HARD のトグルボタンで選択状態を視覚的に表示）
   │ 「スタート」ボタンをクリック/タップ
   ▼
[プレイ画面]
   │ プレイヤーがブラックホールに吸い込まれる or 隕石に衝突
   ▼
[ゲームオーバー画面]
   │ 「もう一度遊ぶ」→ プレイ画面へ（同じ難易度で即リトライ）
   │ 「タイトルへ戻る」→ タイトル画面へ
   ▼
```

### 1-1. タイトル画面（`#screen-title`）
- ゲームタイトルロゴ（テキストで可。CSSでロマ子様コミュニティらしいポップな装飾を施す）
- 難易度選択UI：「かんたん（EASY）」「むずかしい（HARD）」の2択ボタン。選択中のボタンが視覚的にハイライトされる。初期選択は EASY。
- 操作説明（簡潔に1〜2行。例：「矢印キー or 画面タッチでロマ子様を動かして、ブラックホールと隕石を避けよう！」）
- 「スタート」ボタン
- ハイスコア表示（後述のlocalStorageから読み込み。難易度ごとに別々に保存・表示する）

### 1-2. プレイ画面（`#screen-play`）
- `<canvas>` 要素（推奨サイズ 800x600。CSSで画面幅に応じて縦横比を保ったままリサイズ＝レスポンシブ対応）
- 画面上部に合計スコア（生存時間＋かけらボーナス。2-9節）をリアルタイム表示
- canvas内に以下を描画：
  - プレイヤー（ロマ子様。当面は円または簡易アイコンのプレースホルダーでよい。後述）
  - ブラックホール（中心が黒い円。拡大していく）
  - 隕石（複数、ランダムな方向から出現し移動する小円）
  - スターのかけら（黄色く光る星型または小円。収集アイテム）
  - ロマ子様のセリフ吹き出し（特定イベント発生時にプレイヤーの近くに表示）
- 画面上部にダッシュのクールタイム表示（使用可能なら点灯、クールタイム中はゲージや半透明表示など、ひと目で分かるUI）
- スマホ対応：画面下部に簡易の方向ボタン（上下左右、または仮想スティック）を表示。タッチドラッグでも操作可能にする（詳細は4章）

### 1-3. ゲームオーバー画面（`#screen-gameover`）
- 画面切り替えの**前に**、プレイ画面上で「吸い込まれ演出」（2-8節）を約1.5秒再生してから遷移する
- 「GAME OVER」表示＋ロマ子様の捨てゼリフ（2-7節のセリフ集からランダム表示。例：「覚えてらっしゃい…！」）
- 今回のスコア（生存秒数＋かけらボーナスの合計。内訳も小さく表示：「生存 42.3秒 ＋ かけら 8個」）
- ハイスコア（今回更新したら「New Record!」のような表示を出す）
- 「もう一度遊ぶ」ボタン
- 「タイトルへ戻る」ボタン

## 2. コアロジックとルール

### 2-1. 操作方法
- **PC**：矢印キー（↑↓←→）または WASD でプレイヤーを8方向に移動。**スペースキーでダッシュ**（2-6節）
- **スマホ**：画面下部の十字方向ボタン（タップ＆ホールドで継続移動）、もしくは canvas 上でのタッチドラッグでプレイヤーが指の方向に移動（実装が楽な方を選んでよいが、両方実装できるならベター）。**ダッシュ用の専用ボタン**を画面右下に配置
- プレイヤーの移動速度は一定（`PLAYER_SPEED`、後述のパラメータ表参照）。斜め移動の速度が直線移動より速くならないよう正規化すること（ベクトルを正規化してから速度を掛ける）

### 2-2. ブラックホールの挙動
- ゲーム開始時、canvas中央（または中央付近のランダム位置）にブラックホールが半径 `BH_INITIAL_RADIUS` で出現
- 時間経過とともに半径が `BH_GROWTH_RATE`（px/秒）で線形に拡大し続ける
- **引力効果**：プレイヤーとブラックホール中心との距離に応じて、プレイヤーをブラックホール中心方向へ引っ張る力を毎フレーム加算する
  - 引力の強さは `BH_GRAVITY_STRENGTH` で定義し、距離が近いほど強くなる（例：`force = BH_GRAVITY_STRENGTH / max(distance, 50)`、係数はパラメータ表で調整）
  - プレイヤーの最終位置 = プレイヤー自身の移動入力ベクトル + 引力ベクトル を合成して移動させる
  - これにより「ブラックホールに近づくほど操作が効きにくくなる」緊張感を演出する
- **吸い込みモード（引力サージ）**：一定間隔で、短時間だけ引力が大幅に強くなるイベントを発生させる（ゲームが単調にならないよう緊張と緩和のリズムを作る要素）
  - `SURGE_INTERVAL` 秒ごとに発動。発動の `SURGE_WARNING` 秒前から**予兆演出**を出す：画面全体を小刻みに揺らし（canvas描画全体に±3px程度のランダムオフセット）、ブラックホールの縁を赤紫色に光らせる
  - 予兆の後、`SURGE_DURATION` 秒間だけ引力係数を `SURGE_MULTIPLIER` 倍にする。サージ中はブラックホールの見た目も変える（縁の光を強くする、周囲に吸い込み線を描く等、簡易表現でよい）
  - サージ中は隕石の軌道も強く曲がる（2-3節の任意実装を入れている場合）
  - サージ終了で引力は即座に通常値へ戻り、次のサージまでのカウントがリセットされる
  - 初回のサージはゲーム開始から `SURGE_INTERVAL` 秒後（開始直後には発動しない）

### 2-3. 隕石（障害物）の挙動
- 一定間隔（`METEOR_SPAWN_INTERVAL` 秒ごと）で、canvas外周のランダムな位置から隕石が出現
- 出現後、ランダムな方向（ただしcanvas内に向かうおおよその方向）へ一定速度 `METEOR_SPEED` で直進する
- canvas外に完全に出たら配列から削除する（メモリリーク防止）
- 隕石もブラックホールの引力の影響を受けて軌道が中心側にやや曲がると、より自然な見た目になる（任意実装。必須ではない）

### 2-4. 当たり判定
- すべて円同士の衝突判定（AABBではなく円判定を使う。理由：プレイヤー・ブラックホール・隕石いずれも円形のため計算がシンプルになる）
- 判定式：2つの円の中心間距離 < 半径の合計 なら衝突
  ```
  distance(A, B) < (radiusA + radiusB)
  ```
- 以下のいずれかが発生したら即ゲームオーバー：
  1. プレイヤー中心とブラックホール中心の距離 < (プレイヤー半径 + ブラックホール半径)
  2. プレイヤーといずれかの隕石が衝突
- プレイヤーとスターのかけらが衝突（重なった）場合はゲームオーバーではなく**収集**として処理する（2-5節）
- プレイヤーが画面外に出ないよう、canvas境界でプレイヤー座標をクランプする（画面端で止まる。ゲームオーバーにはしない）

### 2-5. スターのかけら（収集アイテム）
リスク＆リターンの核となる要素。**わざとブラックホールの近くに出やすくする**のが最重要ポイント。

- 一定間隔（`STAR_SPAWN_INTERVAL` 秒ごと）に1個出現。画面内の同時存在数は最大 `STAR_MAX_COUNT` 個
- **出現位置の決め方**：ブラックホール中心から距離 `BH半径 + 60px` 〜 `BH半径 + 200px` のドーナツ状の範囲内でランダムに決定する（canvas外にはみ出す場合は再抽選）。これにより「かけらを取りに行く＝ブラックホールに近づく」構図を必ず作る
- かけらは出現後その場に静止。ただしブラックホールの拡大に飲み込まれたら（かけら中心とBH中心の距離 < BH半径）消滅する
- 出現から `STAR_LIFETIME` 秒経っても取られなかったら点滅して消える（点滅は残り2秒間）
- プレイヤーが触れると収集：`STAR_BONUS` 点を加算し、収集数をカウント。取った瞬間に小さなキラッと光るエフェクト（円が広がって消える程度の簡易表現でよい）を出す
- 描画：黄色〜金色の星型（描くのが面倒なら光る小円に十字の光条でも可）。半径 `STAR_RADIUS`

### 2-6. ダッシュ（緊急回避）
「吸い込まれそう→ダッシュで脱出！」の爽快感を作る要素。

- 発動：スペースキー（スマホはダッシュボタン）
- 効果：**現在の移動入力方向**へ `DASH_SPEED_MULTIPLIER` 倍速で `DASH_DURATION` 秒間だけ高速移動する。移動入力がない（ニュートラル）状態で押した場合は、ブラックホール中心と**逆方向**へダッシュする（緊急脱出として直感的なため）
- ダッシュ中も引力・当たり判定は通常どおり有効（無敵にはしない）
- クールタイム：使用後 `DASH_COOLDOWN` 秒間は再使用不可。画面上部のUIに回復状況を表示（ゲージが溜まっていく、またはアイコンの色が戻る等）
- ダッシュ中はプレイヤーの後ろに残像（半透明の円を数フレーム分残す）を描くと爽快感が出る（任意）

### 2-7. ロマ子様のセリフ演出
キャラクターゲームとしての愛嬌を出す要素。実装は「イベント→セリフ配列からランダムに1つ選んで吹き出し表示」のシンプルな仕組みでよい。

- 表示方法：プレイヤーの頭上に吹き出し（白背景・角丸の小さなボックス＋テキスト）を約2秒間表示。表示中に新しいイベントが起きたら上書きする
- セリフはイベント種別ごとに**配列で定義**し、ファイル冒頭の定数部にまとめること（コミュニティで後からセリフを差し替え・追加しやすくするため）
- イベントと初期セリフ案：

| イベント | 発動条件 | セリフ例（配列からランダム） |
|---|---|---|
| 危険接近 | プレイヤーとBH表面の距離が80px未満になった瞬間（一度出たら5秒間は再発動しない） | 「ちょっ、引っ張らないでくださる!?」「吸わないでちょうだい！」 |
| かけら収集 | スターのかけらを取った瞬間 | 「いただきですわ✨」「当然の報酬ですわね」 |
| ダッシュ発動 | ダッシュした瞬間 | 「失礼しますわよ！」「ロマ子様、緊急回避！」 |
| サージ予兆 | 吸い込みモードの予兆演出が始まった瞬間 | 「な、なんか嫌な予感がしますわ…」「来ますわよ…！」 |
| 長生き達成 | 生存30秒・60秒到達時 | 「まだまだ余裕ですわ」「宇宙もわたくしの前ではこの程度」 |
| ゲームオーバー | 死亡演出後、ゲームオーバー画面に表示 | 「覚えてらっしゃい…！」「こ、今回は引き分けですわ…」 |

- セリフの文言は上記をそのまま初期値として実装してよい（後から差し替え前提のプレースホルダー）

### 2-8. ゲームオーバー演出（吸い込まれアニメーション）
悔しさより笑いが先に来る演出にする。

- 衝突検知したら即座にゲームオーバー画面へ飛ばず、**約1.5秒の演出フェーズ**を挟む
- 演出内容：プレイヤー（ロマ子様）がブラックホール中心に向かって螺旋を描きながら吸い込まれていく
  - 毎フレーム、プレイヤー座標をBH中心へ近づけつつ、BH中心を軸に回転させる（角度を毎フレーム加算）
  - 同時にプレイヤーの描画サイズを徐々に縮小し、最後は消える
  - 隕石衝突によるゲームオーバーの場合も同じ演出でよい（衝突→吸い込まれ、で統一するとシンプル）
- 演出中はプレイヤーの操作入力・スコア加算・新規隕石の生成を停止する
- 演出終了後にゲームオーバー画面へ遷移する

### 2-9. スコア計算
- スコア = **生存時間（秒）＋ スターのかけら収集ボーナス**
  - 生存時間：ゲーム開始からの経過秒数（小数点1桁）を1秒＝1点として加算
  - かけらボーナス：1個につき `STAR_BONUS` 点（初期値 5点）
- プレイ画面には「合計スコア」をリアルタイム表示し、ゲームオーバー画面で内訳（生存秒数／かけら個数）を表示する
- ゲームオーバー時のスコアを `localStorage` に難易度別キー（例：`romako_bh_highscore_easy` / `romako_bh_highscore_hard`）で保存し、既存のハイスコアより高ければ更新する

## 3. ファイル構成

```
romako-blackhole-escape/
├── index.html      … 3画面分のHTML構造（タイトル/プレイ/ゲームオーバー）をすべて含む
├── style.css        … 全画面のスタイル、レスポンシブ対応
├── game.js          … ゲームロジック全体（状態管理、入力処理、描画、衝突判定、スコア管理）
└── assets/          … （任意）画像・効果音を追加する場合のフォルダ。なくてもプレースホルダー描画で動作させる
```

- 外部ライブラリ・フレームワーク（React, Phaser等）は使用しない。Vanilla JS + Canvas APIのみで完結させる。
- `assets/` フォルダおよびロマ子様の画像は今回のステップでは用意しない。プレイヤーはCanvas上に `arc()` で描いた円（中にロマ子様を連想させる色、例：ピンク系）でプレースホルダー表示する。後で画像に差し替えやすいよう、プレイヤー描画は `drawPlayer()` のような専用関数に分離しておくこと（画像読み込みへの差し替えが1関数の変更で済むようにする）。

## 4. Codexへの実装指示（フェーズ分け・ロードマップ）

以下のステップを**順番に**実装すること。各ステップの完了時点でブラウザで動作確認できる状態にすること。

### Step 1: HTML/CSSで画面のガワを作る
- `index.html` に3つの画面用div（`#screen-title`, `#screen-play`, `#screen-gameover`）を作成。初期状態は `#screen-title` のみ表示、他は `display: none`
- `style.css` で簡単な見た目を整える（背景色は宇宙っぽい濃紺〜黒のグラデーション、ボタンはポップで丸みのあるデザイン）
- `#screen-play` 内に `<canvas id="game-canvas" width="800" height="600"></canvas>` とスコア表示用の要素を配置
- この段階ではボタンを押しても何も起きなくてよい（次のステップで実装）
- スマホ表示確認のため `viewport` メタタグを設定し、canvasがコンテナ幅に収まるようCSSで `max-width: 100%; height: auto;` 等を指定

### Step 2: 難易度選択と画面切り替えロジックを作る
- `game.js` にシンプルな状態管理（例：`let currentScreen = 'title';` と `function showScreen(name) {...}` で対象div以外を隠す関数）を実装
- EASY/HARDボタンのクリックで選択状態（CSSクラスの付け外しでハイライト）を切り替え、選択中の難易度を変数に保持
- 「スタート」ボタンクリックで `#screen-play` に切り替え、選択された難易度でゲーム初期化関数を呼び出す
- ゲームオーバー画面の「もう一度遊ぶ」「タイトルへ戻る」ボタンの遷移ロジックもここで実装

### Step 3: プレイヤーの移動を実装する
- canvasの2Dコンテキストを取得し、`requestAnimationFrame` によるゲームループを実装
- キーボードイベント（`keydown`/`keyup`）でプレイヤーの移動方向フラグを管理
- プレイヤーの座標を毎フレーム更新し、`drawPlayer()` で円として描画
- canvas境界でのクランプ処理を実装
- この時点ではブラックホール・隕石・当たり判定はまだ実装しなくてよい

### Step 4: ブラックホール（拡大＋引力）を実装する
- ブラックホールオブジェクト（中心座標・半径）を初期化し、毎フレーム半径を拡大
- プレイヤーとブラックホール中心との距離・方向ベクトルから引力を計算し、プレイヤーの移動に合成する（2-2節の式を参照）
- ブラックホールを黒い円＋外周にうっすら紫〜青のグロー（任意）として描画
- **吸い込みモード（引力サージ）**を実装：2-2節の仕様どおり、タイマー管理（待機→予兆→サージ→待機のステートマシン）、予兆の画面揺れ＋縁の発光、サージ中の引力倍率適用を行う

### Step 5: 隕石（障害物）の生成・移動・当たり判定を実装する
- タイマーで一定間隔ごとに隕石を生成し、配列で管理
- 毎フレーム全隕石の座標を更新、画面外に出た隕石は配列から除去
- プレイヤー⇔ブラックホール、プレイヤー⇔各隕石の円判定（2-4節）を毎フレームチェックし、衝突したらゲームオーバー処理を呼び出す

### Step 6: スコアシステムとハイスコア保存を実装する
- ゲーム開始からの経過時間を計測し、プレイ画面上部にリアルタイム表示（この時点では生存時間のみ。かけらボーナスは Step 7 で加算する）
- `localStorage` を使った難易度別ハイスコアの読み込み・保存処理を実装
- ゲームオーバー画面に最終スコアとハイスコア（更新時は New Record 表示）を反映する

### Step 7: スターのかけら（収集アイテム）を実装する
- 2-5節の仕様どおり、ブラックホール周辺のドーナツ状範囲にかけらを出現させる生成ロジックを実装
- プレイヤーとの衝突（収集）判定、`STAR_BONUS` のスコア加算、収集数カウントを実装
- 寿命切れの点滅消滅、ブラックホールに飲み込まれた場合の消滅、同時存在上限を実装
- 収集時の簡易エフェクト（広がって消える円など）を追加
- ゲームオーバー画面のスコア内訳表示（生存秒数／かけら個数）をここで完成させる

### Step 8: ダッシュ（緊急回避）を実装する
- 2-6節の仕様どおり、スペースキーでのダッシュ発動・持続・クールタイムを実装
- ニュートラル状態で押した場合の「ブラックホールと逆方向へダッシュ」の分岐を実装
- 画面上部のクールタイムUI（ゲージまたはアイコン）を実装
- 任意：ダッシュ中の残像描画

### Step 9: ロマ子様のセリフ演出を実装する
- 2-7節のイベント別セリフ配列をファイル冒頭の定数部に定義
- 吹き出しの描画（プレイヤー頭上、約2秒表示、新イベントで上書き）を実装
- 各イベント（危険接近・かけら収集・ダッシュ・長生き達成）の発動条件と再発動制限（危険接近の5秒クールダウン）を実装

### Step 10: ゲームオーバー演出とリトライ動線を仕上げる
- 2-8節の吸い込まれアニメーション（螺旋回転＋縮小、約1.5秒）を実装。演出中は入力・スコア加算・隕石生成を停止する
- 演出終了後に `#screen-gameover` へ切り替え、捨てゼリフ（2-7節のゲームオーバー用配列からランダム）を表示
- 「もう一度遊ぶ」で同じ難易度のまま即座にゲームを再初期化できることを確認
- メモリリークがないか（イベントリスナーの多重登録、配列の肥大化等）を確認し、必要なら `cancelAnimationFrame` で前回のループを確実に止める

### Step 11: 仕上げ・バランス調整・モバイル対応
- スマホでの十字ボタン操作 or タッチドラッグ操作、ダッシュボタンを実装し、実機（またはブラウザのデバイスエミュレーション）で操作感を確認
- 2-10節のパラメータを実際にプレイして微調整し、EASY/HARDで明確な難易度差を感じられるようにする
- バランス確認の観点：「かけらを取りに行く価値があるか（安全プレイより高得点になるか）」「ダッシュで引力から脱出できるか（HARDの引力に対して弱すぎないか）」「サージ中でも画面端付近にいれば生き延びられるか（サージが即死イベントになっていないか）」を必ずチェックする
- 任意：効果音（吸い込まれる音、衝突音、かけら収集音、ダッシュ音）、簡単なパーティクル演出、BGMを追加してもよいが、必須要件ではない

## 5. 実装上の注意事項（Codexへの補足指示）

- 外部ライブラリ・npm・ビルドツールは一切使用しないこと。`index.html` をブラウザで直接開くだけで完全に動作する構成にすること。
- すべてのゲームパラメータ（2-6節の数値）はファイル冒頭等に定数としてまとめ、後から調整しやすい構造にすること。
- 各ステップごとにコードが破綻なく動く状態を保ち、次のステップに進む前に簡単な動作確認ができるようにすること。
- コメントは最小限でよいが、状態管理・衝突判定など複雑なロジックには簡潔な説明を残すこと。
